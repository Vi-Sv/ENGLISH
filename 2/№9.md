# Глава 9. ОПЕРАТОРЫ ПРИОРИТЕТА И ДОСТУПА (МОДАЛЬНЫЕ ГЛАГОЛЫ)

## 1. Введение: Управление правами доступа в синтаксисе
Модальные глаголы — это надстройки над кодом. Они не описывают само действие, а задают права доступа, жесткие ограничения или уровень вероятности его выполнения субьектом.

---

## 2. Инженерия модального синтаксиса: Жесткие правила компиляции
Все базовые модальные операторы подчиняются закону абсолютной чистоты кода:
1. Они **никогда** не принимают окончание `-S` для третьего лица (*Incorrect: He cans*).
2. После них **категорически запрещено** использовать частицу `TO` (*Incorrect: I must to run*). Основной глагол всегда идет в первородной, базовой форме.
3. Они не требуют вспомогательных процессоров `DO / DOES / DID` для отрицания и вопроса.

---

## 3. Матрица операторов и распределение прав (Синтаксис)

### CAN / COULD (Права доступа / Физическая возможность)
*   `CAN` [kæn] — Текущая техническая возможность или физическая способность (Настоящее время).
*   `COULD` [kʊd] — Архивная техническая возможность (Прошедшее время).
*   *Пример:* *I **can** write SQL queries.* (Я могу писать SQL-запросы.) / *We **could not** connect yesterday.* (Мы не могли подключиться вчера.)

### MUST (Абсолютный системный приказ / Hard Constraint)
Высший уровень критичности. Действие обязательно к исполнению, шаг в сторону — обрушение системы.
*   *Пример:* *You **must** change the root password.* (Ты обязан сменить корневой пароль.)

### SHOULD (Рекомендация / Оптимизация кода)
Средний уровень критичности. Совет, регламент или лучшая практика (Best Practice).
*   *Пример:* *We **should** optimize this slow query.* (Нам следует оптимизировать этот медленный запрос.)

### MAY / MIGHT (Уровень вероятности / Дистанционный запрос прав)
*   `MAY` [meɪ] — Запрос официального разрешения или высокая вероятность (*50% True*).
*   `MIGHT` [maɪt] — Низкая вероятность, фоновый шум (*20% True*).
*   *Пример:* ***May** I deploy the script?* (Могу я развернуть скрипт? *Запрос прав*) / *The server **might** reboot.* (Сервер может перезагрузиться. *Низкая вероятность*)

---

## 4. Отрицание и Вопрос: Автономная работа с NOT и Рокировка

### Отрицание (Щит NOT намертво приклеивается к оператору)
*   `can + not` ──> **cannot** (пишется слитно!) / **can't**
*   `must + not` ──> **must not** / **mustn't** (Запрет доступа!)
*   *Пример:* *You **must not** delete the source tables.* (Ты не должен / тебе запрещено удалять исходные таблицы.)

### Вопрос (Вынос оператора на нулевую позицию)
*   *Пример:* ***Should** we run a backup now?* (Нам следует запустить бэкап сейчас?)

---

## 5. Практические примеры
*   **(+)** *The system **can** process big data.* (Система может обрабатывать большие данные.)
*   **(-)** *We **should not** share the credentials.* (Нам не следует делиться учетными данными.)
*   **(?)** ***Could** you fix the bug yesterday?* (Ты мог исправить баг вчера?)
